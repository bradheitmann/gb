# Planning System Guide

**For agents. Read this first. Then read `_STATUS.md`.**

---

## What This Project Is

Read `_NORTH-STAR.md` for the product vision and 12-month outcomes.
This `planning/` directory contains ALL work artifacts: epics, stories, and slices.

## Structure

```
planning/
  _GUIDE.md           You are here
  _NORTH-STAR.md      Product vision (immutable reference)
  _STATUS.md          Auto-generated work index (DO NOT EDIT)
  _templates/         Canonical templates for all artifact types
  EPIC-*.md           Epic definitions (never move, update in place)
  STORY-*.md          Story definitions (never move, update in place)
  epics/              Legacy epic location (being migrated)
  stories/            Legacy story location (being migrated)
  slices/
    backlog/          Defined but not ready (dependencies unmet)
    todo/             Ready to claim (all dependencies met)
    active/           Claimed by an agent (git mv IS the claim)
    done/             Verified complete
    failed/           QA failed (needs remediation)
  holdout/            Story QA test specs (SEALED - see below)
  escalations/        Agent blocker reports (PM reads)
  remediation/        QA failure analysis (PM acts on)
  story-reviews/
    todo/             Auto-created when all story slices done
    active/           Story QA claimed
    done/             Story QA completed
```

## How to Find Your Work

1. Read `_STATUS.md` — it lists everything: available, active, blocked, failed
2. The AVAILABLE NOW section shows slices you can claim immediately
3. Pick by priority (P0 first), then by type matching your role (DEV or QA)

## Dispatch Protocol (5 Steps)

1. **Pull**: `git pull` to get latest state
2. **Read**: Open `_STATUS.md`, find an available slice matching your role
3. **Read slice**: Open the slice file in `slices/todo/`, read fully
4. **Claim**: `git mv slices/todo/SLICE-*.md slices/active/SLICE-*.md`
   - Update `claimed_by`, `claimed_at`, `session_id` in frontmatter
   - `git commit -m "claim: SLICE-ID"` then `git push`
   - If push fails: another agent claimed it. Go back to step 2.
5. **Work**: Create task list using `task_list_id` from slice frontmatter

## What You Must Not Touch

- `holdout/` — Story QA only. Test integrity depends on isolation.
- `_STATUS.md` — Auto-generated by hook. Your edits will be overwritten.
- `_GUIDE.md` — System file. Do not modify.
- `_NORTH-STAR.md` — System file. Do not modify.
- `_templates/` — Read only. Use as reference, don't edit.
- `story-reviews/` — PM and Story QA only.
- Other agents' active slices — Never read or modify.

## When You're Done

1. Ensure all acceptance criteria are met
2. Create evidence bundle at path specified in slice
3. Update slice frontmatter: `status: Done`, `phase: VERIFIED`
4. Move: `git mv slices/active/SLICE-*.md slices/done/SLICE-*.md`
5. Commit and push — the Linear sync hook handles the rest
6. Mark task list tasks as completed

## When You're Blocked

Create an escalation file:
```
planning/escalations/ESC-{SLICE-ID}-{YYYYMMDD-HHMMSS}.md
```
Required fields: `slice_id`, `blocker_description`, `what_was_tried`, `unblock_options`

PM reads escalations and either unblocks you or reassigns.

## When QA Fails a Slice

QA moves slice to `slices/failed/` and creates:
```
planning/remediation/REM-{SLICE-ID}-{YYYYMMDD-HHMMSS}.md
```
PM reads remediation notes and creates targeted fix slices.
Failed slices are never reset — new slices address specific failures.

## Linear Integration

Move the file between folders. That's it. The pre-commit hook detects
the state change and syncs Linear automatically. Never update Linear manually.

## Task Lists

Your slice frontmatter contains `task_list_id`. Use this exact ID when
creating your task list. If resuming a previous session's work on the same
slice, the task list persists — just reconnect to it.

PM's task list ID is set in `.envrc` as `CLAUDE_CODE_TASK_LIST_ID`.

## Communication Rules

| From | To | Channel | Allowed? |
|------|----|---------|----------|
| PM | Dev/QA | Slice file (dispatch) | Yes |
| Dev | PM | Escalation file, folder move, frontmatter | Yes |
| Dev | QA | Never directly | No |
| Dev | Dev | Never (parallel isolation) | No |
| QA | PM | Remediation file, folder move, frontmatter | Yes |
| Story QA | PM | Holdout result via frontmatter + remediation | Yes |

**Why no Dev-to-QA?** QA must evaluate with fresh context. If Dev explains
their reasoning to QA, QA loses independence. QA reads the evidence bundle
and the code — nothing else.

**Why no Dev-to-Dev?** Parallel dev agents sharing reasoning creates
hidden dependencies. Each dev agent works from the slice file only.

## Context Isolation

| Role | Receives | Never Sees |
|------|----------|------------|
| Dev | Slice file, reading list files, codebase | QA slices, holdout/, other Dev reasoning |
| QA | QA slice, evidence bundle, codebase | Dev handoffs, Dev reasoning, holdout/ |
| Story QA | Story file, holdout spec, all done slices | Dev/QA session context |
| PM | Everything | N/A (full visibility) |

## Non-Claude Agents

If your platform doesn't support `git mv` or bash:
1. Update `status:` field in your slice's YAML frontmatter
2. A reconciliation sweep will detect the change and move the file
3. All other rules (context isolation, evidence) still apply
